Rationale

The Tin Can API does a fantastic job of defining communication between learning activities and learner record stores. Sequencing of learning activities to provide a learning path, however, rightly falls outside of Tin Can's scope.

For learning paths including only content produced by the designer of that path, custom sequencing solutions can be provided to link between learning activities. But what if we want to provide a learning path made up of activities from a range of providers? What it we want to include blog posts, YouTube videos and news articles in the mix as well?

For simple learning paths an html page of hyperlinks might suffice. We might even use SCORM to get learner details and launch Tin Can activities from SCOs. Features such as branching or prerequisites based on Tin Can statements would not be possible however. Translating from SCORM to Tin Can is eaay but the reverse is not true and it would be very difficult to feed Tin Can statements of completion or success into SCORM's sequencing and navigation. Branching based on more specialist verbs would be even more impossible.

Another difference with Tin Can from SCORM is that Tin can focuses on achievement and events rather than courses. Could we define a learning map of potential achievements rather than activities?

The goal of this framework is to provide a replacement for SCORM's sequencing functions which is integrated with the Tin Can API. It will provide a way to map out learning paths with branching and prerequisites conditional on Tin Can statements issued by learning activities. It will allow learning designers and curators to pull in eLearning content from a range of suppliers mixed in with other learning activities from the web.

This project is currently at the requirements gathering and ideas drafting stage. Please contribute to the requirements below or email mrdownes@hotmail.com.

Requirements

Target LRS

  • The target LRS endpoint and user credentials should be defined by the LPS (Learning Planning System) rather than the Learning Path Definition
  • These details may vary from user to user or the LPS (Learning Planning System) may have its own credentials with a specific LRS.

Sequencing and Branching

  • The framework should should allow for definition of a learning map (see the 'knowledge map' section of this page: http://www.khanacademy.org/about)
  • The learning map may be a linear sequence or may include branching.

Prerequisites

  • The framework should allow for definition of prerequisites for each node in the learning map.
  • Prerequisites should relate to Tin Can statements
  • The LPS (Learning Planning System) should disable the link from the node to the activity until the prerequisite(s) are met.
    • E.g. you cannot access activity until there is a statement where the actor is the current user, the verb is "completed", the object is activity X and the scaled score is greater than or equal to 0.7.

Approach

Tin Can already specifies a way of planning learning activities using sub-statements of the form '[actor] [planned] [actor] [watched] [Youtube Video]'. We can therefore use a series of these 'planned' statements to build a learning plan. The two main advantages of this approach are:

  1. The framework can reuse Tin Can's definitions
  2. A learner's personal learning plan could be built up and added to with statements from a range of sources at different times.

Verbs

This section lists the verbs used in defining a learning plan and explains how they should be used in this context.

Enrolled

"verb" : { 
 
	"id":"http://www.tincanapi.co.uk/verbs/enrolled_onto_learning_plan", 
 
	"display":{
 
		"en-US":"enrolled onto"
 
		"en-GB":"enrolled onto"
	} 
 
}

This verb is used to create a learning plan or to add additional learners to an existing learning plan. An LPS (Learning Planning System) can obtain a list of learning plans by getting all statements with verb id "http://www.tincanapi.co.uk/verbs/enrolled_onto_learning_plan".

The actor of this statement is the person who is being enrolled onto the plan. Where the enrolment is being assigned by a 3rd party (or the plan was created by a 3rd party), the context instructor property may be used.

The object of statements using this verb will always be an activity. The id of this activity, rather than statement id, should be used to reference the learning plan. For enrolments onto existing activities, the activity definition property is not required as this will already have been defined when the plan was created.

The time-stamp property should be used to define when the enrolment happened, not to set a target completion date for the plan.

After enrolling a new learner on a learning plan, the LPS should reissue all relevant "http://www.tincanapi.co.uk/verbs/planned_learning" statements with the new learner as the actor. The LPS may chose if relevant statements include all statements grouped under the learning plan, or just those issued by the original creator of the plan. The original creator of the plan is defined in the earliest enrolment statement as the context instructor if it exists or the actor if it doesn't.

Where learning plan statements are issued for individual learners on the plan after multiple learners are enrolled on that plan, different implementations of LPS have three options:

  1. Re-issue the statement for all learners enrolled on the learning plan.
  2. Do nothing - the new statement only applies to the learner who issued it and subsequent learners who enrol on the plan (if relevant).
  3. Notify enrolled learners and allow them to chose to accept the change to the plan or not.

Planned

"verb" : { 
 
	"id":"http://www.tincanapi.co.uk/verbs/planned_learning", 
 
	"display":{
 
		"en-US":"planned"
 
		"en-GB":"planned"
	} 
 
}

This verb is used to plan a learning activity, whether this is part of a learning plan or a stand-alone learning goal. The object of this verb should always be a substatement as defined in 4.1.4.3a of the 0.95 spec and explained in my 'I Want This: Tin Can Plans, Goals and Targets' blog post.

The planned item may be included as part of a learning plan, in which case the context contextActivities grouping property should be included with a value of the id of the learning plan.

A prerequisite may be defined using the context statement property. The value of this should be the statement id of an earlier planning statement, that is a statement whose verb id is also "http://www.tincanapi.co.uk/verbs/planned_learning". Each item can only have one prerequisite, but multiple items can share the same prerequisite, forcing a reasonably simple tree structure and preventing hugely complicated webs and impossible loops. If an item requires multiple prerequisites, the activities must be chained one after the other.

For example: 

"context": {
	"statement": "2b71d664-3158-11e2-81c1-0800200c9a66"
	"contextActivities": {
		"grouping" : "http://www.tincanapi.co.uk/wiki/learning_plan:c1356a25-01d0-4301-ba7d-6084c12443d9"
}

Here the prerequisite is an earlier statement whose id was "2b71d664-3158-11e2-81c1-0800200c9a66". This statement belongs to a learning plan "http://www.tincanapi.co.uk/wiki/learning_plan:c1356a25-01d0-4301-ba7d-6084c12443d9".

Cancelled

"verb" : { 
 
	"id":"http://www.tincanapi.co.uk/verbs/cancelled_planned_learning", 
 
	"display":{
 
		"en-US":"cancelled"
 
		"en-GB":"cancelled"
	} 
 
}

This verb is used to record cancelled or amended plans. The object of this verb should always be a statement reference pointed to the id of a "http://www.tincanapi.co.uk/verbs/planned_learning" or "http://www.tincanapi.co.uk/verbs/enrolled_onto_learning_plan" statement.

This verb is a weaker version of "http://adlnet.gov/expapi/verbs/voided" and is used to let the LPS (Learning Planning System) know that this particular plan has been superseded and is no longer relevant. Where a whole learning plan is cancelled, every item in that plan is considered to be cancelled. "http://adlnet.gov/expapi/verbs/voided" would suggest that the plan was never made in the first place.

Where an amendment rather than a cancellation is required, the "http://www.tincanapi.co.uk/verbs/planned_learning" statement(s) should be re-issued with complete with both new values and those which have not changed. Re-issuing a statement will give it a new id, so any statements which depended on the cancelled statement as a prerequisite would also be considered to be cancelled until they are also re-issued.

Completed, Passed and Failed

  • http://adlnet.gov/expapi/verbs/completed
  • http://adlnet.gov/expapi/verbs/passed
  • http://adlnet.gov/expapi/verbs/failed

These verbs can be used with "StatementRef" pointing at a "http://www.tincanapi.co.uk/verbs/enrolled_onto_learning_plan" statement. All of these verbs used in this context assert that the learning plan has been completed. "http://adlnet.gov/expapi/verbs/passed" asserts that the learning plan has been a success. "http://adlnet.gov/expapi/verbs/failed" asserts that the learning plan has been unsuccessful (i.e. learning has not been achieved). "http://adlnet.gov/expapi/verbs/completed" makes no assertion either way about the success of the plan, just that it is over.

The LPS can use its own judgement on when to issue these statements. Some implementations may automatically make learning plans as completed, passed or failed based on fulfilment of the planned learning. Other implementations may require the learner themselves or a supervisor to evaluate and assert the completion and success of the plan.

Defining a template learning plan

This approach works well if we want to produce a learning plan for existing individual or groups of learners, but what if we can to create a template that can be assigned to learners and groups in future? Some points to consider:

  • The actor would need to be removed from the statements or a placeholder actor included
  • id, stored and timestamp properties would also be blank
  • Prerequisite links would need to be preserved despite the lack of ids
  • the verb of the parent statement would always be the same so this need not be stored in the template.

Option 1

A template learning plan might look something like this: 

{
	"id": "0",
 
	"object": {
 
		"objectType": "SubStatement",
 
		"verb" : { "id":"http://example.com/read", "display":{"en-US":"read"} },
 
		"object": {
 
			"id":"http://example.com/book",
 
			"definition": { "name" : {"en-US":"Some Awesome Book"}}
 
		}
 
	}
 
}
 
{	
	"id": "1",
 
	"object": {
 
		"objectType": "SubStatement",
 
		"verb" : { "id":"http://example.com/read", "display":{"en-US":"read"} },
 
		"object": {
 
			"id":"http://example.com/book2",
 
			"definition": { "name" : {"en-US":"Some Other Awesome Book"}}
 
		}
 
	}
 
	"extensions": {
		"http://tincanapi.co.uk/wiki/extensions/learningPlanPrerequisite": "0"
	}
 
}

This template would need to be stored not in the LRS, but in a separate learning plan database.

actor, verb, id, stored and prerequisite properties would be added when the template was applied to the learner or group of learners.

Option 2

Another option would be to create the learning plan for a template actor and then enrol real learners to that plan.
