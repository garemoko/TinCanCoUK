Converting SCORM packages to Tin Can API

This guide provides guidelines on how you might convert e-learning content which currently using SCORM into TCAPI content and suggests how particular SCORM calls might translate into Tin Can statements. The aim is to define a common way of doing this so that all Tin Can traditional e-learning content makes comparable statements at comparable points in the learner's experience. This guide is not intended to be a complete guide for writing Tin Can API compatible content, and if you are new to Tin Can API, it is recommended that you attempt a simpler, single statement activity before progressing further with this guide.

In a lot of places, this guide will tell you what to do without going into detail of how to actually do it. You will need to look up the relevant Tin Can features as they are mentioned if you are not familiar with them. It may be that I produce further guides in future that I can link to from the relevant points in this guide.

This guide will refer to a wrapper for Adobe Captivate swfs which have been published for SCORM (1.2 or 2004). This wrapper intercepts the SWF's attempts to make SCORM calls and then re-interprets them for Tin Can. It's still under development at the moment, but you can get the most recent code here: https://github.com/garemoko/tinCaptivate . You can see a working development version here: http://garemoko.github.com/tinCaptivate/

Fewer calls, longer statements

One major difference between SCORM and Tin Can is that whilst SCORM sends each datum as a single API call, Tin Can groups data into statements, so much more is sent to the LRS in one go. This means sending a much smaller number of API calls. One reason for this is that SCORM reports statuses and Tin Can tends to report events. For example In SCORM we tell the LMS whether or not the learner has passed at a given point in time, whereas in Tin Can, we report the event of the learner passing.

Caching data

Given that you're going to be making fewer calls, you are likely not to want to send data as soon as you get it. This is especially true if you are developing a wrapper, as the SCORM course will make a whole string of calls one after the other and you'll only want to send a statement after the last one comes in. One simple way of doing this is to have an object which stores all of the SCORM data, which can then be used when building a Tin Can statement to be sent.

The Captivate Wrapper example uses an object called "value_store", so when the swf tries to set cmi.score.scaled, the wrapper stores this as value_store["cmi.score.scaled"]. When a statement is being built later, this value is included as the scaled property of the score within the results.

There may be many different approaches to caching data, but the important point from this section is don't send a new statement whenever you get new data. We'll cover when you should send statements in part 3 .

Tin Can's 'State'

Tin Can uses statements for any information you want to report to a human (i.e. the learner or a stakeholder). You may also have data that you want to save for a particular user on the LRS, but which will only be used by your application. The SCORM equivalent of what I'm talking about here is things like the cmi.suspend_data. For these types of things we should be using the State.

Tin Can's State is a lot more flexible than anything SCORM provides in this area in that it allows you to have an many items in the store as you like and also allows you to name these items. If under SCORM, your suspend data was a huge long string of concatenated variables, you should consider using the State to split these up.

If you're building a wrapper, there's not a lot of point in splitting up your suspend data, as you're just going to have to bundle it back up again when the SCO requests it. The Captivate Wrapper simply stores the suspend data as "cmi.suspend_data" in the state.

For wrappers, it is recommended that you store everything that the SCO sends in the state as well as any usage in statements. This allows you to quickly pull back the datum required when the SCO makes a GET request in a later session.

In the 0.95 version of the Captivate wrapper I've handled the state very differently from the 0.9 version. I now store all of the SCORM data as a JSON array in a single document in the State. This is got at the start of the session and stored in the cache for any Get calls made by the Captivate swf. The whole array is then saved once every time the learner changes slide. This has greatly reduced the number of calls to the State and massively improved performance.

When I get time I will expand this section and probably split it off as a separate page in this guide.

The Difference Between Sessions and Attempts

It is easy to get confused between attempts and sessions, so as these concepts will be important for the next few sections of the guide, lets first make sure we're on the same page.

An attempt is one trip through the e-learning content. If for example the content is a quiz, an attempt is working through all the questions from first to last. An attempt may be completed all all in one go, or in parts. I might answer questions 1-5 one day and then answer the remaining questions on another occasion. It is also possible to have multiple attempts in a row without leaving the e-learning package.

A session is one time when the learner access the e-learning. It doesn't matter if they don't attempt any questions or complete the quiz 4 million times. The session starts when the user navigates to the content and ends when they leave the page or close the window.

In SCORM terms, a session is the period between Initialize and Terminate and an attempt is the learning between completions.

Initialization and termination

SCORM uses initialization and termination to handle sessions. In Tin Can, there is no requirement to use sessions, and statements can be sent at any time to report passes, failure, completions etc. Many SCORM style e-learning courses will wish to use sessions, and for this we can use the "http://adlnet.gov/expapi/verbs/initialized" and "http://adlnet.gov/expapi/verbs/terminated" verbs to mark the end of the session.

The 'initialized' verb should be reported at the start of the session. The 'initialized' statement should report a result duration of "PTHM0S" as in the statement snippet below: 

"verb": {
    "id": "http://adlnet.gov/expapi/verbs/initialized",
    "display": {
        "en-GB": "initialised",
        "en-US": "initialized"
    }
},
"result": {
    "duration": "PTHM0S"
}


The 'terminated' verb should be reported at the end of a session. This might be when the user clicks an exit button, but it's also recommended to try and catch the browser window's close event as well. The 'terminated' statement should report a result duration of the session duration. Note, this is different from the attempt duration (which we will discuss in the next section) and is identical to SCORM's cmi.session_time. The statement snippet below provides an example. 


"verb": {
    "id": "http://adlnet.gov/expapi/verbs/terminated",
    "display": {
        "en-GB": "terminated",
        "en-US": "terminated"
    }
},
"result": {
    "duration": "PT3M11S"
}


In order to handle situations where the 'terminated' verb is not sent at the end of a session (e.g. the user closes the browser window and the close event dopes not fire) the following rules should be followed:

1. If an 'initialized' statement is reported when there is already an open session, then a reporting tool can assume that the previous session ended at the timestamp of the last reported statement for that user and activity. Statements where the activity is listed within the contextActivities should also be considered in making this calculation.

2. If a session is left open, but not further 'terminated' or 'initialized' statements are reported, the reporting tool may calculate session duration based on the time-stamp of the last statement received. Further statements may lead to this duration being increased.

3. If a 'terminated' statement is received but there is no open session, it will be stored by the LRS but will be ignored by reporting tools in calculating session duration.

If you're building a wrapper the SCO may make initialization and termination calls. Since the 'initialize' is normally called when the SCO loads, you may wish to ignore any initialize calls and simply begin the session when the page loads. You may still need to still catch the call and return whatever the SCO is expecting to say that initialize was successful. The Tin Can Captivate Wrapper, for example, returns "true" when either initialize or terminate are called. You may need to provide a way of capturing and handling terminate calls, as these may be sent, for example, from an exit button.

Duration

SCORM provides mechanisms for reporting both session time and attempt time, using cmi.session_time and cmi.total_time respectively. It is recommend to track and report both these items if using sessions in Tin Can. If not using sessions and it is relevant and practical to do so, attempt time should still be tracked and reported. We have already seen that session time should be reported with the 'terminated' statement. The next section will cover statements where attempt time should be reported. These statements are those used to report passed, failed, completed and suspended verbs.

Some SCORM packages may only report cmi.session_time (session duration) or cmi.total_time (attempt duration) and expect the LMS to work out the other figure. The Captivate SCO only reports cmi.session_time, so the wrapper calculates the cmi.total_time every time the cmi.session_time updates and stores it in the state. If you're interested in the exact mechanism for this and can't work it out from the code, please get in touch.

Functions for converting between CMITimespan format and seconds as well as for adding or subtracting CMITimespan formatted durations are provided in The Tin Can Captivate Wrapper's TinCanCan.js and these may be copied and used freely so long as the copyright notice at the top of the js file is placed with them and the terms of this notice are accepted.

Completion, Success

SCORM uses cmi.success_status and cmi.completion_status to track success and completion. We mentioned earlier that Tin Can does not report success and completion status, but instead tracks completion and success events. When converting SCORM to Tin Can API, you therefore need to look for changes in cmi.success_status and cmi.completion_status and report these changes as statements. In a wrapper, this can be achieved by caching cmi.success_status and cmi.completion_status then comparing them with the values sent by the SCO. When there is a difference in value, it is time to send a statement, though you may need to wait until the SCO finishes passing all relevant data (see section Identifying groups of calls in part 2 of this guide).

The values of cmi.success_status and cmi.completion_status determine exactly which verb is sent to the LRS to mark the end of the an attempt. This is shown in the table below:

cmi.completion_status cmi.success_status verb sent to LRS
completed passed passed
completed failed failed
completed unknown completed
incomplete any suspended
not attempted / unknown any none

See [| Part 3] for a reference table of when to send each statement and the verb ids for each verb.

'passed', 'failed', and 'completed' statements all mark the end of an attempt. Following this type of statement, any further statements should be considered part of a new attempt.

Quiz results

Tin Can statements have a 'result' property which you should use to send quiz results at the end of an attempt. The items within 'result' have fairly obvious SCORM equivalents. In regards to score, the Tin Can API recommends sending the scaled score wherever possible.

Unlike SCORM's cmi.success_status and cmi.completion_status, "success" and "completion" in Tin Can are just true or false. You will need to do some sconversion to translate SCORM to Tin Can. The following table shoos the conversion:

cmi.completion_status cmi.success_status Tin Can equivalent
completed passed true
incomplete failed false


Remember also, the duration in 'passed', 'failed', and 'completed' statements is attempt duration rather than session duration, i.e. cmi.total_time

Interactions

Interactions in SCORM tend to be things like quiz questions. In Tin Can API, these questions are considered to be child activities of the main activity. You should use different verbs for interactions . 'interacted' (verb id: http://adlnet.gov/expapi/verbs/interacted) can be considered to be equivalent of 'experienced'. 'answered' (verb id: http://adlnet.gov/expapi/verbs/answered) is equivalent to 'completed'. The Captivate Wrapper always uses the verb 'answered' because it reports answered questions only. It does not have a concept of session for questions, though a duration is provided using SCORMS's cmi.interactions.n.latency.

The Interaction Object

Statements about interactions need to have their own object, rather than simply using the parent course object. The activity type 'http://adlnet.gov/expapi/activities/cmi.interaction' should be used as it can include the properties 'interactionType' (equievelent to SCORM's cmi.interactions.n.type) and correctResponsesPattern (an array of strings equivalent to cmi.interactions.n.correct_responses.n.pattern).

Captivate does not seem to pass an interaction description (this has not been thoroughly tested) so the Captivate wrapper has to create one.

Interaction Objects should be linked to their parent activities using the statement's context property as described in the spec. Here's an example modified from a statement generated by one of Rustici's prototypes: 

"context": {
    "contextActivities": {
        "parent": {
            "id": "scorm.com/GolfExample_TCAPI/GolfAssessment.html",
            "definition": {
                "name": {
                    "en-US": "Golf Example Assessment"
                },
                "description": {
                    "en-US": "An Assessment for the Golf Example course."
                },
                "type": "http://adlnet.gov/expapi/activities/assessment"
            },
            "objectType": "Activity"
        },
        "grouping": {
            "id": "scorm.com/GolfExample_TCAPI",
            "definition": {
                "name": {
                    "en-US": "Golf Example - Tin Can Course"
                },
                "description": {
                    "en-US": "An overview of how to play the great game of golf."
                },
                "type": "http://adlnet.gov/expapi/activities/course"
            },
            "objectType": "Activity"
        }
    }
}

The Interaction Result and Timestamp

The result of the interaction statement is also slightly different from the parent SCO's result. We still need to convert the success into a true or false and then also need to calculate the score for the question. This is done by giving the learner a 0 if they failed the question and the value of"cmi.interactions.n.weighting if they passed. If you want to do things like partial credit, you'll need to consider a different way of reporting this.

The response is simply SCORM's cmi.interactions.n.learner_response and the duration is cmi.interactions.n.latency. The timestamp for this statement is cmi.interactions.n.timestamp 

var interactionResult = getResult(true);
interactionResult.response = value_store["cmi.interactions." + interactionIndex + ".learner_response"];
interactionResult.duration = value_store["cmi.interactions." + interactionIndex + ".latency"]
//Score
interactionResult.score = new TinCan.Score;
interactionResult.score.min = 0;
interactionResult.score.max = parseInt(value_store["cmi.interactions." + interactionIndex + ".weighting"]);
switch (value_store["cmi.interactions." + interactionIndex + ".result"])
{
	case "correct":
		interactionResult.success = true;
		interactionResult.score.scaled = 1;
		interactionResult.score.raw = interactionResult.score.max;
	break;
	case "incorrect":
		interactionResult.success = false;
		interactionResult.score.scaled = 0;
		interactionResult.score.raw = interactionResult.score.min;
	break;
	default:
	break;
}

Slides as child activities

Depending on the level of tracking you want to implement, you may wish to report every slide/page/screen that a learner visits. As with interactions, it's advisable not to try to use sessions for these small chunks. Of course, if you are doing something a bit more significant on a page, you may want to treat it as a full activity with sessions, attempts etc. In that case, you're reading the wrong section of the guide!

Because slides don't tend to be discreet attempts in their own right with passing, failure or completion, it is recommended to report slide views with the 'experienced' verb. Each slide will need it's own object definition and it is recommended that the id be based on the coruseId as in the interactions example above. There is no need to pass any result, but the timestamp can still be included.

As with interactions above, the context should be used to relate child activities to their parents.

Intercepting SCORM calls

When building a wrapper for a SCORM based swf, you need to catch when the swf tries to make a SCORM call. There is likely to be some JavaScript function in the existing SCORM html wrapper which the swf calls in order to communicate with the LMS. We therefore need to re-write or overwrite this function to communicate with the Tin Can LRS instead.

Adobe Captivate's default wrapper contains 2 such functions: Captivate_DoFSCommand and Captivate_DoExternalInterface

In Captivate 5.0 and 5.5 (the most recent version at time of writing), it appears that only Captivate_DoExternalInterface actually gets called. The Tin Can Captivate Wrapper replaces that function with the following: 

function Captivate_DoExternalInterface(command, parameter, value, variable)
{
	varstrErr = "true";
 
	myTinCan.log("Captivate_DoExternalInterface: command: "+command+" parameter: "+ parameter+" value: "+ value+" variable: "+ variable);
 
	//TODO: Check if we have a connection to the TCAPI, if not: return;
 
	//We don't want to do anything when Captivate calls Initialize, Terminate, Commit or GetLastError. We only need handle get and set with TinCan
	 if ( command == "LMSSetValue" || command=="SetValue") {
		strErr = TCCSetParameter(parameter,value);
 
	} else if ( command == "LMSGetValue" || command=="GetValue") {
		strErr = TCCGetParameter(parameter, variable);
	}
	else  if ( command == "LMSGetLastError" || command=="GetLastError")
	{
		strErr = 0;
	}
	else  if ( command == "LMSTerminate" || command=="Terminate")
	{
		endSession();
 
	}
	else
	{
		strErr = "true";
 
	}
 
	CaptivateObj.SetScormVariable(variable, strErr);
	return strErr;
}

Notice how we are only really handling SetValue and GetValue calls (or LMSSetValue and LMSGetValue for SCORM 1.2). For every other type of call we simply return "true", which tells Captivate that its command has executed successful, even though we've done nothing! This is because the wrapper itself handles initialization and termination etc. This could be amended if you wanted to do something when your swf tried to initialize or terminate.

In the case of LMSGetLastError or GetLastError, we always return 0. This tells Captivate that there have been no errors.

The line CaptivateObj.SetScormVariable(variable, strErr); is simply lifted from the existing SCORM wrapper. In the case of setting values it returns whether or not the call was successful to a specific captivate variable. In the case of getting values, it sets a specified captivate variable to the result of the GET.

In designing your Tin Can wrapper, it is important to take time studying and following through the logic of the existing SCORM wrapper. Make sure you are returning exactly the same values as the swf expects to receive from the SCORM wrapper. Using an LMS with a debug mode, such as SCORM Cloud will allow you to keep watch exactly what values are being passed to and fro between the SCO and LMS.

Identifying groups of calls

We mentioned in the section above, Fewer calls, Longer Statements, the need to cache SCORM data and send it all together in statements. This section will look at one way of identifying that the SCORM content has stopped sending data before we send the statement.

With SCORM, one event tends to trigger changes in multiple variables and rapid e-learning software therefore tends to report groups of data in quick succession. Adobe Captivate, for example sends batches of data after initialize, when the slide changes and when a quiz question is answered. We can therefore watch for these events and send the statement once all the data for that event has come through. Captivate also sends data in the same order every time. This means we can know when a batch of data is complete by watching for the last item of data associated with that batch.

If you are building a wrapper for another rapid e-learning tool, perhaps it will work in a similar way. You can watch what statements a SCORM Object is outputting and when by plugging it into an LMS with a debugging tool and watching the log as you progress through the content.

Let's take the example of sending the completion statement in The Tin Can Captivate Wrapper. This statement needs to be sent when the completion statement changes from incomplete or complete, but not until the batch of data in which completion status changes is done. Captivate always reports cmi.location, cmi.session_time then cmi.suspend_data after it reports cmi.completion_status. This means if we sent the statement as soon as cmi.completion_status changed, we'd be missing out on the updated session time.

Here's the code: 

case "cmi.completion_status": 
 
if((compareWithCacheAndSetState(parameter,value,false)) && (value == "completed")) {
	completionChanged = true;
}					
 
break;

and 

case "cmi.suspend_data": //Captivate's Suspend Data string 
if (completionChanged)
	{
		completionChanged = false;
		TCCSendLessonData();
	}
break;

When the SCO changes cmi.completion_status, we change completionChanged to true. Then, when cmi.suspend_data is set later we change it back to false and send the statement. completionChanged is a global variable initially set to false.

Handling GET calls

So far we've been looking at sending data to the LRS, but when building a wrapper the SCO will also request data from the LMS. This section will look at how you should return that data to the SCO.

Using the State

The first type of data that your SCO will ask for is data which it has previously sent, perhaps in an earlier session. The easiest way to deal with this type of call is to simply store everything that the SCO sends out in Tin Can's state (as described above). You can then pull this out again when the SCO requests it.

Rustici Software's Tin Can Javascript library contains the following functions for getting and setting the state:

  • TinCan.setState
  • TinCan.getState

See [1] for examples of how these may be used.

Responding to this type of GET is therefore fairly straightforward. The items this applies to are:

  • cmi.location
  • cmi.suspend_data
  • cmi.score.scaled
  • cmi.score.min
  • cmi.score.max
  • cmi.score.raw

Sometimes you already know the answer

Another type of GET call is where the SCO wants to find out some information about the LMS, for example which features are supported or getting some information from the manifest file. Since we are writing the wrapper, we know what features are supported by our wrapper so can hard code the responses to these gets. Here's the relevant parameters for The Tin Can Captivate Wrapper and what this wrapper returns to its sco:

  • cmi.score._children returns 'scaled,min,max,raw'
  • cmi.interactions._children returns 'id,type,objectives,timestamp,correct_responses,weighting,learner_response,result,latency,description'
  • cmi.launch_data returns

cmi.launch_data, which pulls the value of dataFromLMS from the manifest is not used by most rapid e-learning tools. If you did want to use it, you could either hard code the value or write some code to pull it from an xml or properties file.

cmi.entry

SCORM's cmi.entry is used to tell the SCO whether the learner has accessed the sco before or not. If this is the first time, the LMS will return either an empty string, or 'ab_initio' (which is Latin for 'from the beginning'). If this is not the first time, the LMS returns 'resume'.

Tin Can does not have an equivalent to cmi.entry, so it's down to the wrapper creator to make sure the right value is passed to the SCO. The Tin Can Captivate Wrapper solves this with the following code: 

var entryState = TCCGetFromCache("cmi.entry");
//Next time the user loads this activty, cmi.entry tells captivate that they have accessed this before. 
compareWithCacheAndSetState("cmi.entry","resume",false);
return entryState;
  1. Get the value of cmi.entry from the store. The result of this function will be an empty string if it has not previously been set, i.e. the first time the learner accesses the content.
  2. Set the value of cmi.entry to 'resume' ready for next time the sco requests it.
  3. Return the original value to the sco.
